# Step-4 Training v2 — G-series architectural rebuild

> **Supersedes** the F-series (F1 → F10), closed at F10a on 2026-04-28. F-series investigation record: `step4_training.md`. Operations entry-point (cross-cycle log): `../04_augmentationGSS_hpc.md`.

---

## Aim

Break the F-series structural floor (composite 1.306 at F10a, with AT_HOME, act_JS, and cop_cal_MAE gates still failing) by changing the model and training pipeline rather than running another single-axis hyperparameter trial. Three of the four failing gates were diagnosed in the F-series as architectural — autoregressive feedback amplification (H3), conditioning collapse (H4), and target-stratum sampling bias (H1) — not loss-weight tunable.

The five candidate fixes (consolidated from `step4_training.md` §5 and the external proposal in `newArchProposal.md`):

- **A. Cross-attention conditioning** over a `[cond_vec ‖ cycle_emb ‖ strata_oh]` embedding, replacing additive `strata_linear` + FiLM. Targets H4 (decoder ignores conditioning).
- **B. Scheduled sampling / slot-dropout** on the teacher-forced `dec_aux_seq` AT_HOME channel during training (p=0.2 per slot). Targets H3 (autoregressive stuck-at-home).
- **C. Label smoothing** on the home BCE ({0,1} → {0.05, 0.95}) to prevent sigmoid saturation. Targets H3 (sigmoid > 0.70 lock-in).
- **D. Proportional target sampling** in `04C_training_pairs.py` — sample target strata in proportion to the population, not uniformly. Targets H1 (~2.3× weekend over-representation).
- **E. Decoder capacity expansion** — raise `d_model` and `d_ff`. Cited in F10a closure as a structural ceiling.

**Decision: stage the diffs, but allow bundled submission.** Two-part rule:

- **Diff-bundling — forbidden.** F4 demonstrated the failure mode of bundling unrelated changes (FP16 + new loss + new sampling collapsed into "INVALID, can't attribute"). Each G-stage edits a single axis (one file or one small same-hypothesis cluster of files) so any successful run is attributable.
- **Submission-bundling — allowed for independent stages.** Two stages whose diffs touch disjoint files (e.g. G2 in `04D_train.py`, G3 in `04B_model.py`) may ship in one array submission via the multi-tag sweep mechanism (`tags: [G2, G3]`). Each tag retains its own checkpoint, its own `outputs_step4_G{tag}/`, its own `diagnostics_v4_statistical.json`, and its own row in `results.csv`. This compresses the upload-wait-download cycle without sacrificing per-stage attribution.

**Hard gates (inherited from F-series, evaluated per stage):** composite < 1.045 AND AT_HOME ≤ +5.3 pp AND Spouse ≤ +5 pp AND act_JS ≤ 0.05.

## Steps (task list)

> **Standing rule — Architecture archival.** Every stage in this list (G-series, H-series, and any future stage) that modifies architecture (`04B_model.py` class signatures, decoder/encoder topology, positional encoding, head wrapping, generation path, embedding strategy, loss shape, or training-pair construction) must archive the predecessor file to `Speed_Cluster/archive/<file>_<predecessor_tag>.{py,yaml}` *before* the edit lands, in the same commit. Env-var-gated config knobs over an unchanged codebase do not trigger archival. Full rule and worked H-series examples: see *Architecture archival rule* subsection inside the H-series block below. Reviewer MODE B blocks any architecture edit missing its archive entry.

### G1 — Proportional target sampling

- **Change scope:** `04C_training_pairs.py` only. No model change. No training-loop change.
- **Targets:** H1 (training-pair target-stratum sampling bias, ~2.3× weekend over-representation).
- **What it does NOT touch:** `04B_model.py`, `04D_train.py`, the F10a YAML config, env-var defaults.
- **Independence check:** running F10a's `configs/F10a.yaml` against the new pairs file with `seed=42` produces a single deterministic delta vs. F10a — only the supervision distribution changes.
- **Rationale:** cheapest, highest-confidence diagnosis. The 2.3× weekend over-representation is the most concrete documented cause of the AT_HOME baseline push. If G1 alone clears AT_HOME ≤ +5.3 pp and act_JS ≤ 0.05, no architectural rewrite is needed.
- **Advancement gate:** all four hard gates pass → ship as Step-4 production. Partial pass (any subset of gates closed but not all) → continue to G2.

### G2 — Training-loop interventions (B + C bundled)

- **Change scope:** `04D_train.py` only. (a) scheduled sampling on `dec_aux_seq` AT_HOME with p=0.2 per slot; (b) label smoothing on home-BCE targets {0,1} → {0.05, 0.95}.
- **Targets:** H3 (autoregressive feedback amplification, sigmoid saturation > 0.70).
- **What it does NOT touch:** model architecture (`04B_model.py`), data pipeline (`04C_training_pairs.py`), loss-weight env vars.
- **Why bundled:** B and C target the same failure mode (H3) via the same mechanism (break the train-inference feedback loop). Either succeeding individually still resolves H3. New env vars `SCHED_SAMPLE_P` and `HOME_LABEL_SMOOTH` (defaults 0.0 → backward-compatible with F-series). Setting both to zero must reproduce G1 behaviour exactly.
- **Joint evaluation gate (under bundle-submission):** G2 ships alongside G3 in one array submission (see Bundle-Submission Protocol below) once G2 smoke is green. Both rows are evaluated together against the four hard gates. If G2 alone clears all four, ship G2 as Step-4 production (G3 row becomes a comparison data point). If G2 closes AT_HOME but Spouse / act_JS still fail, the G3 row in the same submission is the answer; conditioning was the residual issue. If neither tag clears the gates, residual is data-quality / schema (see Expected result below).

### G3 — Cross-attention conditioning + decoder capacity expansion

- **Change scope:** `04B_model.py`. Replace `FiLMTransformerDecoder` with a cross-attention path over `dec_cond = [cond_vec ‖ cycle_emb ‖ strata_oh]`. Raise `d_model` (256 → 384) and `d_ff` (1024 → 1536).
- **Targets:** H4 (decoder ignores conditioning) plus the F10a-cited capacity ceiling.
- **What it does NOT touch:** `04C_training_pairs.py`, `04D_train.py` loss/sampling logic (G1 + G2 stay in effect — `SCHED_SAMPLE_P=0.2`, `HOME_LABEL_SMOOTH=0.05`, `data_dir: outputs_step4_G1` all carried into `configs/G3.yaml`).
- **Rationale:** F8 isolation showed AUX_STRATUM_HEAD=1 was the only knob that meaningfully moved AT_HOME (+5.3 → +1.41 pp), confirming stronger conditioning is the right axis — but the aux-head route distorts other channels (Spouse +19.4 pp, act_JS regressed). Cross-attention provides the conditioning signal without sharing trunk capacity. Single-GPU first; only escalate to DDP if the `pg` partition cannot fit.
- **Timing under bundle-submission:** G3 implementation begins as soon as G2 smoke clears green — *not* after G2 full completes. G3 ships in the same array submission as G2-full under tag `G3` (see Bundle-Submission Protocol).
- **Advancement gate:** all four hard gates pass on the G3 row → ship G3 as Step-4 production. Partial pass → see "If both fail" branch in Expected result.

### G4 — Remove scheduled sampling, keep G3 architecture

- **Change scope:** `configs/G4.yaml` only. No source-file changes.
- **Targets:** Spouse regression introduced by `SCHED_SAMPLE_P=0.2` in G2/G3. Secondary: AT_HOME 0.758 pp gap via `lambda_home` bump.
- **What it does NOT touch:** `04B_model.py`, `04C_training_pairs.py`, `04D_train.py`.
- **Rationale:** G3 cross-attention architecture cleared act_JS (0.024 ≤ 0.05) without relying on scheduled sampling — the architecture itself suppresses H3 autoregressive feedback. SCHED_SAMPLE_P=0.2 is now net-harmful: it zeroes AT_HOME teacher-forcing slots to 0.5-neutral, corrupting the Spouse conditional co-presence signal (Spouse can only be home if person is home). G1 (no sched_sample) had Spouse=0.95 pp; G2 (sched_sample on) blew it to 15.39 pp; G3 (same + larger model) worsened it to 19.77 pp. The G3 model at sched_sample=0.0 is the cleanest test of whether architecture-only H3 mitigation holds.
- **Key risk:** act_JS may rise above 0.05 gate without sched_sample. G1 (no cross-attn, no sched_sample) had act_JS=0.068. G4 has G3's cross-attention — plausible that it holds act_JS < 0.05, but unconfirmed.
- **Fallback:** if act_JS fails at sched_sample_p=0.0, resubmit as G4b with sched_sample_p=0.05 (75% reduction from 0.2 — reduces Spouse corruption while retaining partial H3 prevention).
- **Advancement gate:** all four hard gates pass → ship G4 as Step-4 production. act_JS fails → G4b (dial up sched_sample_p to 0.05). Spouse still fails with sched_sample_p=0.0 → revisit schema / target-definition (data issue, not architecture).

### H-series — prior-paper-grounded architectural alternatives

Pending G4 result. If G4 (config-only, drop `SCHED_SAMPLE_P=0`) closes ≤2/4 gates, the H-series is the next architectural fork. Two improvement proposals (`improveStep4_opus.md`, `improveStep4_gemini.md`) converge on the same root cause: the encoder-decoder/autoregressive formulation creates the H3 problem class that scheduled sampling band-aided and Spouse paid for. The H-series ports specific deltas from the user's prior-paper Transformer (Energy & Buildings, S037877882600215X) — the credibility argument is that exact architecture worked on a harder problem (146 activity classes, 23 demographics) in the same lab.

#### Architecture archival rule (applies to every architecture update — H-series and beyond)

**Permanent operating rule.** Whenever *any* future stage modifies architecture (`04B_model.py` class definitions, decoder topology, positional encoding, head wrapping, generation path, or any equivalent rewrite of `04A_data.py` / `04C_training_pairs.py` / `04D_train.py` that changes the model contract), the **previous version must be archived before the edit lands** so any prior checkpoint or future audit can be replayed without git-archaeology. The archive is what makes the F→G→H→… lineage reproducible — without it, "G3 was a cross-attention decoder" or "H_Tanh added Tanh heads" becomes a claim instead of a runnable artefact. This rule applies to H-series and to every stage that follows (G5, I-series, J-series, …) — not just the H block below.

- **Per-stage archive folder.** Before any class-definition edit, copy the current file to `Speed_Cluster/archive/<file>_<predecessor_tag>.{py,yaml}` (examples: `04B_model_G3.py` before H_Tanh lands; `04B_model_H_Tanh.py` before H_Time lands; `04B_model_H_Time.py` before H1/H_NAT lands). One file per architectural ancestor; never overwrite an existing archive.
- **Config archival pairs with code archival.** When a stage's config schema changes (new env-var gate, new YAML key, new sweep driver), archive the predecessor's `configs/<TAG>.yaml` alongside the model file in the same folder. The archive folder is the answer to "what exactly was *<predecessor>*", not a cleanup workspace.
- **What triggers archival vs. what does not.** Triggers: any change to a `nn.Module` class signature, decoder/encoder topology, head wrapping, embedding strategy, positional encoding, generation path, loss-function shape, or training-pair construction. Does **not** trigger: env-var-gated config knobs over an unchanged codebase (e.g., H4 `LAMBDA_ALL_EQUAL`, H6 `ACTIVITY_BOOSTS`, H3-d_ff once `--d_ff` plumbing has landed) — these are config selections, not architecture updates.
- **H-series archive points (worked examples).** H_Tanh archives G3 (`04B_model_G3.py`). H_Time archives the post-H_Tanh model (`04B_model_H_Tanh.py`) for the PE rewrite. H2 archives the post-H_Tanh model for the embedding rewrite. H1/H_NAT archives the post-Tier-2 model — biggest delta of the series, archive is non-negotiable. H4/H6/H3-d_ff are env-var/config-only and do **not** trigger archival.
- **Diagnostics archival is automatic.** Each tag's `outputs_step4_<TAG>/diagnostics_v4_statistical.json` and its `results.csv` row are archived implicitly by the per-tag output isolation already in the bundle protocol. No additional step required for diagnostics.
- **Builder responsibility.** The Sonnet builder round that ships any architecture edit must perform the archival copy as the *first* step of the diff and include it in the same commit. Reviewer (MODE B) flags any model-class edit landing without the matching archive folder entry as a hard block, regardless of whether the edit is H-series or post-H.

#### Tier 1 — Bundle-able cheap ablations

Designed so all three H-stages ship alongside G4 in **one** four-tag sweep submission. Each is a single-axis edit; together with G4 they collapse four serial upload/train/download cycles into one.

- **H_Tanh — bound `home_head` and `cop_head` with `nn.Tanh()`** before the linear projection. From `improveStep4_gemini.md` §2. Targets sigmoid saturation on the binary heads (the F-series H3 sub-symptom that label smoothing has been masking). Scope: `04B_model.py` two-line edit (heads). Effort: S. Bundle compatibility: edits `04B_model.py` only — disjoint from `04D_train.py` config knobs of H4/H6. Advancement gate: clears AT_HOME and Spouse vs. G4 row → carry Tanh forward into Tier 2.
- **H4 — equal 1:1:1:1:1 loss weights, normalized per batch.** From `improveStep4_opus.md` §3. Targets the F-series hand-tuned `(λ_act, λ_home, λ_cop, λ_marg, λ_spouse) = (1.0, 0.5, 0.3, 0.1, 0.1)` over-fit risk. Scope: env-var gate `LAMBDA_ALL_EQUAL` in `04D_train.py`; when set, override config lambdas to 1.0 and divide by Σλ at batch-loss time. Config: `configs/H4.yaml` (G4 base + `lambda_all_equal: 1`). Effort: S. Bundle compatibility: env-var-gated → config-only at submission time, disjoint from H_Tanh's model-class edit and H6's class-weight edit. Advancement gate: any gate moves favourably vs. G4 → diagnostic positive, weight tuning was a contributor.
- **H6 — drop activity-class boosts and inverse-sqrt-frequency weighting.** From `improveStep4_opus.md` §3. Targets the F-series hand-calibrated `Work +5.0 / Transit +3.0 / Social +2.0` boosts as an entrenched bias the gates penalize. Scope: env-var gate `ACTIVITY_BOOSTS` in `04D_train.py`; when set to 0, set CrossEntropy `weight=None`. Config: `configs/H6.yaml` (G4 base + `activity_boosts: 0`). Effort: S. Bundle compatibility: same env-var pattern as H4, lives in a different `04D_train.py` block. Advancement gate: act_JS holds < 0.05 and AT_HOME improves → boosts were net-harmful.

**Tier-1 bundle hard requirement.** The two `04D_train.py` env-var flags (`LAMBDA_ALL_EQUAL`, `ACTIVITY_BOOSTS`) must be plumbed *before* the bundle ships. Tiny edits (≤10 lines each) but load-bearing — without them, H4 and H6 would both modify the same `04D_train.py` blocks and re-introduce the F4 attribution problem.

#### Tier 2 — Medium-effort architectural moves (gated on Tier-1 result)

- **H_Time — learnable positional encoding + explicit cyclical time features.** From `improveStep4_gemini.md` §3+§4. Replace fixed `sinusoidal_pos_enc` with `nn.Parameter(torch.randn(1, 48, d_model) * 0.02)`; concatenate `[sin(2πt/48), cos(2πt/48)]` cyclical features into decoder slot inputs. Targets human-behavior punctuation at exact hours (8 AM, 5 PM commute) that smooth sinusoids can't capture. Scope: `04B_model.py` (PE block + decoder input assembly). Effort: M. Bundle compatibility: edits same file as H_Tanh — must ship sequentially or as a separate solo tag (cannot bundle with H_Tanh in the same submission unless H_Tanh has already landed and H_Time builds on it).
- **H2 — per-feature `nn.Embedding(card, k)` for categorical demographics.** From `improveStep4_opus.md` §3. Replace the pre-computed one-hot demographic vector with learnable embeddings (one `nn.Embedding` per categorical column, concatenated and projected to d_model). Targets H4 (decoder ignores conditioning) at the embedding layer. Scope: `04A_data.py` (stop one-hot encoding at dataset assembly) + `04B_model.py` (add `ModuleList` of embeddings + projection). Effort: M. Bundle compatibility: edits two files; cannot bundle with H_Time (both touch `04B_model.py`); can bundle with H_Tier-1 stragglers if any are still in flight.

#### Tier 3 — Reserve structural rewrite (gated on Tier-1 + Tier-2 still leaving gates open)

- **H1 / H_NAT — encoder-only / non-autoregressive prediction.** Both proposals converge here. Drop the decoder entirely (or remove its causal mask + token feedback); predict all 48 slots in one parallel pass from encoder output. Eliminates H3 at the architectural level → no SCHED_SAMPLE needed → Spouse cannot regress from slot-zeroing. Scope: new `EncoderOnlyOccupancyModel` (or NAT variant) in `04B_model.py`; `04D_train.py` drops scheduled-sampling and label-smoothing plumbing. Configs: new `configs/H1.yaml` deriving from G1 (proportional sampling carries; G2/G3 knobs become no-ops). Effort: L. Bundle compatibility: ships solo. Pre-flight: do NOT run if G3/G4 + Tier-1/2 close 3/4 gates — never throw away a near-pass.
- **H3-d_ff — wide `d_ff` × shallow depth sweep.** From `improveStep4_opus.md` §3. Sweep `(n_layers, d_ff) ∈ {(2, 8192), (3, 4096), (4, 2048)}` at d_model=256. Scope: `04D_train.py` `--d_ff` argparse plumbing (currently hard-coded as `d_ff = d_model × 4`) + `D_FF` mapping in `config_to_env.{sh,py}` + sweep YAML. Effort: S after `--d_ff` plumbing lands. Bundle compatibility: once plumbed, three arms ship as one three-tag bundle (config-only differences over a shared codebase).

#### H-Tier-1 Bundle Protocol

- **One sweep YAML.** `configs/sweep_H_tier1.yaml` with `tags: [G4, H_Tanh, H4, H6]`.
- **Four config files.** `configs/G4.yaml` (already specified in G4 task block), `configs/H_Tanh.yaml`, `configs/H4.yaml`, `configs/H6.yaml`. All four derive from G3 base + the single axis change for that tag.
- **One smoke driver.** `configs/sweep_smoke_H_tier1.yaml` with the same four tags at `--sample` mode. All four smokes must clear green before any `pg`-slot full submission.
- **Output isolation.** Each tag writes to `outputs_step4_{G4,H_Tanh,H4,H6}/`, its own `diagnostics_v4_statistical.json`, and its own row in `results.csv`. Comparison against the G3 row remains row-by-row in the same CSV.
- **Calendar.** ~24h smoke-to-rows for all four tags vs. ~4 days if shipped sequentially. Single upload/wait/download cycle.
- **Hard rule preserved.** Bundle works because the four diffs are: G4 = config-only, H_Tanh = `04B_model.py` heads, H4 = `04D_train.py` lambda block (env-var gated), H6 = `04D_train.py` activity-weight block (env-var gated). The two `04D_train.py` arms touch *different* code blocks, both env-var-gated → independent config selections, not overlapping diffs. F4 attribution rule satisfied.
- **Pre-bundle landing requirement.** Sonnet must plumb `LAMBDA_ALL_EQUAL` and `ACTIVITY_BOOSTS` env vars *before* the bundle ships. These two plumbing edits land alongside the H_Tanh model-edit in a single `04*` source-file commit; the four config YAMLs land separately.

### Bundle-Submission Protocol

Once G2 smoke is debugged green, G2-full and G3-full ship as a *single* array submission, not two sequential submissions. Mechanics:

- **One sweep YAML, two tags.** `configs/sweep_G23.yaml` with `tags: [G2, G3]`. The existing `submit_step4_array.sh` → `job_04D_train_array.sh` machinery iterates over tags; each tag's `04D` job loads its own `configs/${TAG}.yaml` via `config_to_env.sh`. Two independent training jobs run in parallel under one job-array index space.
- **Per-tag config files.** `configs/G2.yaml` (G1 base + `sched_sample_p: 0.2`, `home_label_smooth: 0.05`, `data_dir: outputs_step4_G1`) — already exists. `configs/G3.yaml` (G2 base + cross-attention arch params + same `data_dir: outputs_step4_G1`) — to be written when G3 implementation lands.
- **Per-tag smoke driver.** `configs/sweep_smoke_G23.yaml` with `tags: [G2_smoke, G3_smoke]` — both smokes must clear green before any `pg` slot is paid for.
- **Output isolation.** Each tag writes to its own `outputs_step4_G{2,3}/`, with its own `diagnostics_v4_statistical.json` and its own row in `results.csv`. Comparison against F10a / G1 baselines remains row-by-row in the same CSV.
- **The hard rule.** Bundle-submission only works because G2 (`04D_train.py`) and G3 (`04B_model.py`) touch disjoint files. Two tags whose diffs *both* edit the same file would re-introduce the F4 attribution failure and must remain sequential.

## Expected result

- **G1:** ~60 % subjective probability of clearing AT_HOME on its own; lower for closing all four gates simultaneously. The `obs_home_rate` shift alone (toward population-weighted ~0.65 from uniform-stratum-weighted ~0.725) directly attacks the diagnosed cause.
- **G2:** independently closes the residual AT_HOME gap if G1 doesn't fully close it, by killing the autoregressive home-state lock-in.
- **G3:** primary mechanism for closing Spouse + act_JS simultaneously, since both are downstream of conditioning collapse (H4).

If all three stages run and the four gates remain unmet, the residual is a data-quality / target-definition issue, not an architecture issue, and Step-4 work pivots to revisiting the schema (e.g., is `obs_home_rate ≈ 0.725` realistic given the `IS_SYNTHETIC=0` AT_HOME column construction).

## Test method

1. **G1 reproducibility check.** Bit-identity against F10a is *not* expected (the training pairs change). Instead: `seed=42`, log `obs_home_rate` from the new pairs, confirm it shifts toward population-weighted ~0.65 from uniform-stratum-weighted ~0.725.
2. **Per-stage attribution.** Each stage produces its own `diagnostics_v4_statistical.json` under `outputs_step4_G{1,2,3}/`. Comparison row appended to `results_index/results.csv` via the existing `extract_metrics.py`. Side-by-side audit against F1, F8, F10a in the same CSV.
3. **Backward-compat.** Default env vars must reproduce F10a. `SCHED_SAMPLE_P=0`, `HOME_LABEL_SMOOTH=0`, and the unchanged `04C` output file must yield F10a's `step4_training_log.csv` byte-for-byte (modulo float noise).
4. **No diff-bundling.** Each tag edits a single axis (one file or one same-hypothesis cluster). *Submissions* may be bundled via multi-tag sweeps (G2 + G3 in one array) — that does not violate this rule because the diffs remain attributable per-tag in `results.csv`. Two tags whose diffs both edit the same file remain sequential.
5. **Smoke first on every tag, including bundled tags.** Use the existing `configs/sweep_smoke*.yaml` pattern — 1 epoch, single GPU, ~5 min per tag — before paying for any `pg` slot. Under bundle-submission both smokes (`G2_smoke` and `G3_smoke`) must clear green before either tag is shipped to `pg`.
